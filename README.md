Reflection
1. Briefly summarize the DriverPass project. Who was the client? What type of system did they want you to design?
The client for this project was DriverPass, a company focused on improving student success on DMV driving tests. They wanted a cloud‑based system that allows students to register, access online learning materials, take practice tests, schedule on‑the‑road lessons, and track their progress. The system also needed tools for staff roles such as the owner, IT officer, secretary, and instructors to manage accounts, appointments, reports, and DMV updates.

2. What did you do particularly well?
I did especially well in translating the interview transcript into clear, structured requirements and ensuring that each requirement mapped directly to a system function. My UML diagrams were also strong, especially the use case and class diagrams, which clearly showed how the system components interact. I also excelled at documenting constraints, assumptions, and role‑based access needs in a way that made the system design easy to understand.

3. If you could revise one part of your work, what would you improve?
If I were to revise anything, I would expand the technical requirements section of the SDD. While it met the assignment expectations, I could improve it by adding more detail about security architecture, scalability planning, and how DMV updates would be processed automatically. This would make the design more realistic for a production‑level system.

4. How did you interpret the user’s needs and implement them into your system design? Why is this important?
I interpreted user needs by analyzing the interview transcript, identifying pain points, and converting them into functional and nonfunctional requirements. Each requirement was then represented in the UML diagrams and system components.
Understanding user needs is essential because a system that doesn’t align with real workflows will fail, even if it is technically sound. Designing around the user ensures the system is intuitive, efficient, and solves the actual problems the client described.

5. How do you approach designing software? What techniques or strategies would you use in the future to analyze and design a system?
My approach begins with requirements gathering, followed by modeling the system using diagrams that clarify structure and behavior. I rely on:

Use case diagrams to understand user interactions

Activity diagrams to map workflows

Sequence diagrams to show system communication

Class diagrams to define architecture

Role‑based access analysis to ensure security and permissions

Iterative refinement to adjust the design as requirements evolve

In the future, I will continue using user‑centered design, modular architecture, and clear documentation to create systems that are maintainable, scalable, and aligned with business goals.
